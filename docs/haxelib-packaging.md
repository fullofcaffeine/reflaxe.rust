# Haxelib Packaging Notes

This repo packages release zips with `scripts/release/package-haxelib.sh`.

## Goals

- Keep packaging behavior aligned with Reflaxe `build` conventions.
- Preserve runtime/compiler assets required by this target (`runtime/`, `vendor/`).
- Avoid std override leakage into non-target contexts.

## Source and Packaging Model

Source checkouts follow the layout generated by `haxelib run reflaxe new`:

- `src/**` contains compiler/framework code.
- `std/**` contains Rust-target APIs, hxrt externs, and native support modules.
- `std/rust/_std/**/*.hx` contains Rust replacements for upstream stdlib modules.

The checked-in `haxelib.json` keeps that contract explicit:

```json
"reflaxe": {
  "name": "Rust",
  "abbv": "rs",
  "stdPaths": [
    "std",
    "std/rust/_std"
  ]
}
```

Release zips are built by delegating the generic layout work to the vendored Reflaxe `Run build`
runner:

- It reads `classPath` and `reflaxe.stdPaths` from `haxelib.json`.
- It copies `classPath` into the package.
- It merges each `stdPath` into the packaged `classPath`.
- It converts files copied from paths ending in `_std` to `.cross.hx`.
- It copies release files and sanitizes `haxelib.json` by removing Reflaxe metadata.

The script then adds target-required `runtime/` and `vendor/` assets that generic Reflaxe packaging
does not know about.

## Reflaxe conventions followed here

This repo should behave like a normal Reflaxe compiler unless a Rust-specific asset requires extra
packaging:

- source std/support paths are declared in `haxelib.json` `reflaxe.stdPaths`;
- source-mode tests use those paths before Haxe typing starts;
- package builds use Reflaxe `Run build` for classpath flattening, `_std` -> `.cross.hx` conversion,
  release-file copying, and `haxelib.json` sanitization;
- target-specific release logic is limited to copying `runtime/` and `vendor/` after Reflaxe has built
  the haxelib-shaped package.

Do not add a checked-in or symlinked `src/**` mirror of `_std` source files. That would make the
source tree imitate generated package output and reintroduce resolution-order ambiguity that Reflaxe's
`stdPaths` model already solves.

## Source Checkout Development

For repo-local tests and examples, use the checked-in `haxe_libraries/reflaxe.rust.hxml`. It supplies
both `std/` and `std/rust/_std/` as explicit initial classpaths, matching Reflaxe's dev/test flow.

If a source-checkout consumer is not using this repo's haxe library file, it must add the same source
paths explicitly:

```hxml
-cp path/to/reflaxe.rust/src
-cp path/to/reflaxe.rust/std
-cp path/to/reflaxe.rust/std/rust/_std
```

Do not use raw `haxelib dev reflaxe.rust <checkout>` as the source-checkout validation path for this
repo. Plain haxelib dev only puts the package `classPath` on the initial classpath, while
source-checkout std overrides live under `std/rust/_std`. Macros in `extraParams.hxml` run too late to
guarantee ownership of every upstream-colliding std module. `haxe.Exception` is the concrete failure
mode: if upstream `haxe.Exception` is typed before `_std` is visible, generated Rust can lose the
backend-owned typed exception module even though the target macros later run.

For installed packages, consumers should just use:

```hxml
-lib reflaxe.rust
```

The installed package already contains the generated `src/**/*.cross.hx` files produced by Reflaxe
build, so consumers do not add `_std` paths themselves.

## Why `.cross.hx` still appears in packages

Checked-in upstream-colliding std overrides live as `_std/*.hx` source files. Packaged zips expose those
same overrides as `.cross.hx` files under `src/**`, because Reflaxe build turns `_std` source paths into
target-conditional package files.

The compiler should not infer semantic module identity by parsing those package filenames. Haxe source
positions are still file paths, so the compiler needs path roots for ownership checks, but module
identity should come from Haxe/Reflaxe typed metadata (`ModuleType.getModule()` / `getPath()`).

## Validation workflow

Use `bash scripts/ci/package-smoke.sh` to validate the shipped artifact end-to-end:

- Build the zip with `scripts/release/package-haxelib.sh`.
- Verify package layout + metadata invariants (`src/` flattening, sanitized `haxelib.json`, pruned runtime artifacts).
- Compile a minimal app from the source checkout using repo `haxe_libraries` to confirm `_std` source overrides work without generated `.cross.hx` files.
- Create an isolated local haxelib repo (`haxelib newrepo`) and install the zip.
- Compile a minimal app with `-lib reflaxe.rust` and confirm std override modules are emitted.
- Compile again from a symlinked working-directory alias to exercise path canonicalization behavior.
- Build the generated Rust crate with `cargo build`.

Important: validate packaged behavior through `haxelib install` + `-lib reflaxe.rust`, not raw `-cp <pkg>/src`.
Raw classpath-only tests are not equivalent for packaged `.cross.hx` std override selection.

## Stdlib provenance guards

Packaging checks are complemented by stdlib governance guards:

- `node scripts/ci/upstream-stdlib-boundary-check.js`
  - enforces `vendor/haxe/**` remains untracked,
  - restricts tracked std override file types under `std/`.
- `node scripts/ci/stdlib-provenance-ledger-check.js`
  - enforces `docs/stdlib-provenance-ledger.json` coverage for tracked `std/rust/_std/**/*.hx` files.

## Backend-specific requirement

The compiler resolves runtime assets relative to library root (`runtime/hxrt`) and bootstrap classpaths
for vendored Reflaxe (`vendor/reflaxe/src`), so both directories are part of the distributed artifact.
