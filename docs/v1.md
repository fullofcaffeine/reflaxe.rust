# v1.0 Support Matrix (Graduation Contract Scope)

For the current evidence-backed status table, use `docs/feature-support-matrix.md`.
This page stays focused on the broader v1 contract and release scope.

Important:

- this document defines the proposed `1.0` contract scope,
- the canonical current release line is generated from `release-manifest.json` and reported in
  [Semver And Release Posture](semver-release-posture.md),
- it does not by itself prove that every listed surface is broadly production-closed today,
- current proof depth still comes from the support matrix plus the linked CI/test evidence.

This document defines what “v1.0” means for `reflaxe.rust`: which Haxe features and std modules are
considered supported, which profile contracts exist, and what the default build workflow looks like.

v1.0 is intentionally strict: if something is not listed here, treat it as “may compile, but not
part of the stability contract yet”.

## Build workflow (native by default)

Minimal compile:

- `haxe -lib reflaxe.rust -D rust_output=out -main Main`

Defaults:

- Generates a Cargo crate in `out/` (with `hxrt` runtime bundled).
- Runs `cargo build` best-effort after codegen (debug profile).

Opt-outs / knobs:

- Codegen-only: `-D rust_no_build` (alias: `-D rust_codegen_only`)
- Release build: `-D rust_release` (alias: `-D rust_build_release`)
- Run rustfmt after codegen: `-D rustfmt`
- Choose Cargo subcommand: `-D rust_cargo_subcommand=build|check|test|clippy|run`
- Pass target triple: `-D rust_target=<triple>`
- Force nullable string wrapper mode: `-D rust_string_nullable`
- Force legacy non-null string mode: `-D rust_string_non_nullable`

## Profiles

Profiles are compile-time flags that trade “Haxe portability” vs “Rust-first performance”.

Set via:

- `-D reflaxe_rust_profile=portable|metal`

### Portable (default)

- Prioritizes Haxe semantics and portability.
- Uses runtime wrappers when Rust ownership would otherwise conflict with Haxe expectations
  (example: `Array<T>` aliases on assignment).
- String representation defaults to nullable `hxrt::string::HxString` (Haxe-compatible `null` support).

### Metal

- Rust-first mode for teams that need typed low-level interop in selected places.
- Includes Rust-first helper surfaces (`rust.*`) and performance-oriented defaults.
- Keeps `rust.metal.Code.expr(...)` / `rust.metal.Code.stmt(...)` as scoped raw bridge APIs,
  not as normal app-facing facades.
- Enables strict app-boundary policy by default (raw app-side `__rust__` is rejected).

## “No injection in apps” rule

Policy:

- Apps/examples should not call `__rust__` directly.
- Put injections behind Haxe APIs in `std/rust/*` (escape hatch still exists, but should be rare).

Enforcement:

- Examples and snapshot tests compile with `-D reflaxe_rust_strict_examples`.

## Supported language features (v1.0 target)

This list is the minimum surface expected to remain stable for v1.0:

- Expressions: literals, locals, assignment, unary/binary ops
  - Compound assignment includes String class-field append (`field += value`) with Haxe assignment
    result semantics.
- Equality: `==` / `!=` for objects and arrays uses Haxe identity semantics
- Control flow: blocks, if/else, while, for-range, for-iter
- Functions: calls, closures/function values
  - Includes `this.method` closures, reusable callback forwarding/storage, and mutable captured-local callback paths
    on the documented portable contract (see `docs/function-values.md`).
- Types: classes, interfaces, inheritance (portable dispatch model)
  - Inheritance includes method inheritance shims (subclasses get concrete Rust methods for base methods).
  - `super.method(...)` is supported (compiled via per-base “super thunk” methods on the current class).
  - Uninitialized concrete class-reference fields default to the runtime null `HxRef<T>` handle. Codegen must not
    recursively call the field class constructor just to produce a placeholder value.
- Properties (`var x(get,set)` / `var x(default,set)`):
  - Non-storage properties are not emitted as struct fields.
  - Property reads/writes lower to `get_x()` / `set_x(v)` calls.
  - Assignment result matches Haxe semantics: `obj.x = v` returns the setter return value (not always `v`).
  - Storage-backed accessors (`default,get` / `default,set`) treat `x` inside `get_x`/`set_x` as direct backing-field
    access to avoid recursive calls.
- Enums + `switch` lowering to Rust `match`
- Exceptions: throw + try/catch
- Generics: common cases used by std and Rust-first helpers
  - Generated class/interface type parameters default to `Clone + Send + Sync`.
  - Generic helper methods propagate generated class payload bounds when their signatures mention
    bounded payloads, without adding runtime indirection.

Language-feature caveats:

- No separate language-feature blocker is tracked on this page right now.
- Use the [Known limitations](#known-limitations-v10-non-goals) section for explicit defers.
- Use the [Feature Support Matrix](feature-support-matrix.md) for current proof depth and package-level status.

## Core stdlib + overrides

The Rust target ships a small Rust runtime crate (`runtime/hxrt`) and overrides a subset of the Haxe
stdlib in `std/` to ensure predictable semantics.

Notable items:

- `Array<T>` maps to `hxrt::array::Array<T>` (a lightweight wrapper around `HxRef<Vec<T>>`).
  - `HxRef<T>` is the runtime's nullable shared handle for Haxe reference values, backed by thread-safe
    ownership/interior-mutability primitives (so values can safely cross OS threads).
  - Generic nullable std results such as `Array<Class>.shift()` map empty results back to the
    reference type's explicit null handle when the Haxe type is `Null<Class>`.
- `String.substring(start, end)` lowers through compiler-owned Haxe bounds logic plus the existing
  string slicing helper. Haxe's `start > end` swap behavior is covered by
  `test/snapshot/string_substring`.
- `haxe.io.Bytes` maps to a runtime buffer type (`hxrt::bytes::Bytes`)
- Compiler emission model:
  - By default, codegen emits user project modules plus this repo's `std/` overrides.
  - Upstream Haxe stdlib modules are type-checked but not emitted unless you opt in with
    `-D rust_emit_upstream_std`.
- `Sys.getEnv(name)` follows the upstream nullable contract (`Null<String>`):
  - missing keys return `null`
  - present keys return the environment value
  - behavior is covered by `test/snapshot/sys_getenv_null`

## Reflection subset (portable)

Portable v1.0 includes a deliberately small reflection surface (compile-time where possible):

- `Type.getClassName(TypeExpr)` and `Type.getEnumName(TypeExpr)` for `TTypeExpr` inputs.
- `Reflect.hasField(obj, "field")` for constant field names on:
  - class instances (`TInst`): checks declared instance fields/methods
  - anonymous objects (`TAnonymous`): checks structural fields
  - runtime dynamic objects (`Dynamic` backed by `DynObject` / `Anon`)
- `Reflect.field(obj, "field")` for constant field names on:
  - class instances: supports instance `var` fields (boxed into `Dynamic`)
  - anonymous objects: supports structural fields (boxed into `Dynamic`)
  - runtime dynamic objects (`Dynamic` backed by `DynObject` / `Anon`)
- `Reflect.setField(obj, "field", value)` for constant field names on:
  - class instances: supports instance `var` fields
  - anonymous objects: supports general record literals (lowered to `hxrt::anon::Anon`)
  - runtime dynamic objects (`Dynamic` backed by `DynObject` / `Anon`)
- `Reflect.compare(left, right)` for typed `Int`, `Float`, and `String` values. This lowers at
  compile time to direct Rust equality/greater-than checks that return Haxe-style `0`, `1`, or `-1`;
  `test/snapshot/reflect_compare_sort` covers use inside an `Array.sort` comparator.

Notes:

- This is not “full RTTI reflection”. If the field name is not a compile-time string literal, the call is unsupported.
- `Reflect.compare` on unsupported/non-basic value types fails as an unsupported Rust-target
  lowering instead of emitting a phantom `Reflect::compare` Rust call.
- `Reflect.setField` accepts either a concrete typed value or a `Dynamic` value that can be downcast to the target field type at runtime.
- General anonymous record literals lower to `HxRef<hxrt::anon::Anon>` so Haxe aliasing and mutation semantics survive across helper returns and constructor arguments. Required typed fields are read through the declared structural type; `test/snapshot/anon_required_field_types` covers a Bool/String/string enum-abstract/Int record, including a `functionNumber:Int` field fed by helper and JSON-value decoding paths. Optional anonymous fields omitted at construction are guarded by generated `has_key(...)` checks and fall back to the field type's Haxe null representation; `test/snapshot/anon_optional_fields` covers omitted `String` and `Bool` fields read later.

## Rust-first API layer (selected)

The `std/rust/*` package provides Rust-leaning APIs that remain “Haxe shaped”:

- Ownership/borrows: `rust.Ref<T>`, `rust.MutRef<T>`, `rust.Borrow.withRef/withMut`
- Collections: `rust.Vec<T>`, `rust.HashMap<K,V>` (+ Tools)
- Slices: `rust.Slice<T>`, `rust.MutSlice<T>`, `rust.SliceTools`, `rust.MutSliceTools`
  - Works on `rust.Vec<T>` and (in metal profile) on `Array<T>` without cloning.
- Option/Result: `rust.Option<T>`, `rust.Result<T,E>` (+ Tools)

## Known limitations (v1.0 non-goals)

- Full Rust lifetime modeling: Haxe has no lifetime syntax; borrow helpers must keep borrows scoped.
- Zero-cost output everywhere: correctness and predictability beat perfectly minimal output in Portable.
- Full stdlib parity is an active execution track, not a closed gate:
  - portable parity scope follows `docs/stdlib-policy.md` (cross-target std APIs such as `Std`, `haxe.*`, `sys.*`)
  - target-specific namespaces (`cpp.*`, `js.*`, `lua.*`, `php.*`, `python.*`, `hl.*`, `neko.*`) and backend-native `rust.*` are out of parity scope
  - progress is tracked via Tier1 snapshot/sweep coverage and stdlib governance guards
- Typed catch follows emitted non-generic class and interface subtype chains (see
  `test/semantic_diff/typed_catch_subclass` and `test/semantic_diff/typed_catch_interface`).
  Concrete payloads are downcast and upcast to the annotated class/interface, while values already
  boxed in a polymorphic representation retain an exact typed fallback. Generic classes and paths
  without emitted subtype metadata still fall back to exact `Dynamic.downcast` behavior.
- `sys.thread.EventLoop` direct operations are covered on the Rust target, and higher-level target-side Rust smoke evidence now covers both the basic `haxe.MainLoop.add(...)` + `haxe.EntryPoint.run()` path (`test/snapshot/haxe_mainloop_entrypoint_basic`) and the thread-bridge path using `haxe.MainLoop.addThread(...)` + `haxe.MainLoop.runInMainThread(...)` (`test/snapshot/haxe_mainloop_entrypoint_thread_bridge`). Broader `haxe.MainLoop` / `haxe.EntryPoint` semantics still rely on narrower target-side evidence rather than broad `--interp` semantic parity.
- `rust_async` is supported on a documented Rust-first subset, not a broad blanket async contract across profiles/runtime modes.
- `Dynamic` boxing requires `Clone`: values passed through `hxrt::dynamic::from(...)` (printing, exception
  payloads, some std APIs) must be Rust-`Clone`. Most Haxe values are (`String`, `HxRef<T>`, `Array<T>`, etc.),
  but some extern/native types may need a wrapper API.
