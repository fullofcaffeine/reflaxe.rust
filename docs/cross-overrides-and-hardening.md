# `.cross.hx`, `_std`, and Family Hardening Notes

This document records how `reflaxe.rust` currently uses Reflaxe `_std` source overrides and generated
`.cross.hx` package files, what that means for sibling coexistence, and where hardening still makes sense.

## Current model in this repo

`reflaxe.rust` uses `std/rust/_std/**/*.hx` as its checked-in stdlib override source model.

This repo intentionally does **not** have an early checked-in `src/haxe/*.cross.hx` mirror. Source
mode uses `_std` through Reflaxe-style `stdPaths`; package mode gets `.cross.hx` from Reflaxe build.

Bootstrap activation is also relatively narrow:

- `target.name == "rust"`
- or `-D rust_output=...`

That is an important difference from broader Haxe 4 `Cross` activation patterns.

## Quick matrix

| Question | Answer for this repo |
| --- | --- |
| Main override style | `std/rust/_std/**/*.hx` as source; packaged `src/**/*.cross.hx` generated by Reflaxe build |
| Is `_std` used? | yes, as the checked-in upstream-override source lane |
| Is `.cross.hx` used broadly? | yes, but as generated package output |
| Does this repo own early `src/haxe/*` modules? | no |
| Should source dev add a package-output mirror? | no; use `stdPaths`/`_std` before typing starts |
| Bootstrap activation currently keys off raw Haxe 4 `Cross`? | no |
| Same-compilation sibling-target coexistence safe today? | not guaranteed |
| Highest-priority hardening item | add mixed-target fail-fast while preserving narrow target activation |

## What `_std` means here

In this repo, `_std` is the normal checked-in source lane for upstream-colliding std overrides.

That follows the layout generated by `haxelib run reflaxe new`:

- `std/` contains target APIs and support modules.
- `std/rust/_std` contains Rust replacements for upstream stdlib modules.
- repo-local dev/test hxml files put `std/rust/_std` on the initial classpath.

That last point is not optional for source-checkout workflows. Haxe can type core std modules before
target init macros run, so macro-time classpath injection cannot reliably replace an initial `_std`
classpath.

## What `.cross.hx` means here

`.cross.hx` is the generated packaged target-override model.

Reflaxe build copies `std/rust/_std/**/*.hx` into the packaged `src/**` classpath as
`*.cross.hx`. The checked-in source tree should not carry generated package mirrors.

## Current coexistence risk

This repo does not have the same early `src/haxe/*` collision profile as OCaml and Elixir.

That lowers the risk.

But it still shares many module names with sibling targets, including:

- `StringTools`
- `Date`
- `Lambda`
- `Sys`
- `haxe.Exception`
- `haxe.CallStack`
- `haxe.Constraints`

The most important one is `haxe.Exception`.

Rust owns that module under `std/rust/_std/haxe/Exception.hx`, which becomes
`src/haxe/Exception.cross.hx` in a built package. OCaml and Elixir currently own early
`src/haxe/Exception.cross.hx` files.

If a sibling early file wins resolution first in a mixed-target compile, Rust can lose the real implementation it expected.

## Risk level

Current status:

- default one-target-at-a-time use: acceptable
- same-compilation multi-target coexistence: still risky enough to deserve hardening

This repo is not the highest-risk member of the family, but it should not rely on classpath order if mixed activation becomes more common.

## Hardening direction

Recommended next steps:

1. Keep bootstrap activation narrow and target-specific.
2. Add explicit mixed-target detection/fail-fast behavior when conflicting sibling target libraries are active.
3. Document `haxe.Exception` ownership as a family collision surface that should not silently resolve by classpath luck.
